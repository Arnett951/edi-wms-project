EDI 940 · WMS Integration — CR Pipeline

Scope vs. shipped: what V0 actually cost

The original design for the human-governed autonomous delivery pipeline was scoped at roughly 3.9M–6.5M tokens across 45–72 sessions, spread over ten phases plus integration hardening. What actually shipped as a working V0 — SQL-backed change requests, three real human gates, automated implementation and merge/rollback dispatch — landed at roughly 150K–200K tokens across 3 sessions. This is where that gap came from, phase by phase.

Original scope
3.9M–6.5M tok
45–72 sessions, 10 phases + hardening
Shipped V0
<200K tok
3 sessions, 3 phases full + hardening
Reduction
~95–97%
by cutting scope, not by cutting corners on what shipped

Phase-by-phase

  1. 1. Chat intake & requirements clarification
    est. 300k–500k tok
    Shipped
    What shipped
    Claude-driven chat intake that clarifies scope, classifies Tier A/B/C, and writes a structured change request — built essentially as scoped.
    What was cut
    Nothing significant. Closest phase to the original spec.
  2. 2. Change request store & approval workflow
    est. 250k–400k tok
    Shipped
    What shipped
    SQL-backed CR table (survives redeploys), status-driven approve/reject UI, three real gates instead of one.
    What was cut
    No separate audit-log table — each CR keeps only its current status line (approver + timestamp), not a full history of every transition.
  3. 3. Spec / acceptance criteria / test cases / deploy plan
    est. 350k–550k tok
    Cut
    What shipped
    The CR schema itself carries requirements, touch points, and out-of-scope notes — a lightweight spec, not a separate document.
    What was cut
    No dedicated acceptance-criteria doc, no generated test-case list, no deploy plan. Folded entirely into the coding agent's own judgment.
  4. 4. Coding-agent orchestration
    est. 800k–1.4M tok
    Shipped
    What shipped
    Headless Claude Code, isolated git worktrees, automated branch/commit/push, an actual budget cap. Self-healing within the run itself — the agent writes tests, sees a failure, and fixes it before finishing. The one phase built close to spec.
    What was cut
    Guardrails are a tool allowlist + budget cap, not a hardened sandbox.
  5. 5. PR packaging & automated checks
    est. 500k–900k tok
    Cut
    What shipped
    A one-click "Approve & Merge" that runs a real merge and pushes.
    What was cut
    No pull request is ever opened — straight merge to main. No dedicated security scan, lint gate, or regression suite; verification rests on whatever the agent did while implementing.
  6. 6. Dev/staging deploy automation
    est. 150k–300k tok
    Cut
    What shipped
    Production-only auto-deploy on every merge to main.
    What was cut
    No staging tier, no versioning, no promotion step — every test happens directly against production. Flagged as the next real gap to close.
  7. 7. Notifications
    est. 100k–200k tok
    Cut
    What shipped
    Nothing.
    What was cut
    No email/Teams/Slack integration at all. Admin checks the CR-Workflow tab manually.
  8. 8. UAT capture & formal approval
    est. 200k–300k tok
    Cut
    What shipped
    The Gate 2 human click stands in as the final sign-off.
    What was cut
    No distinct UAT environment, no stakeholder click-through, no formal sign-off record separate from the merge click itself.
  9. 9. Production promotion (versioned package)
    est. 350k–550k tok
    Cut
    What shipped
    Nothing beyond the standard zip-deploy already in place.
    What was cut
    No artifact versioning, no blue-green or slot-swap — each merge overwrites the live app directly.
  10. 10. Monitoring & rollback
    est. 350k–550k tok
    Partial
    What shipped
    Real, automated rollback — one click reverts the merge commit and pushes, dispatched via GitHub Actions when needed.
    What was cut
    No App Insights / Azure Monitor alerting. Self-healing stops at deploy time -- there's no post-deployment equivalent, so rollback is human-triggered because a person noticed something wrong, not monitor-triggered.
  11. Integration & end-to-end hardening
    est. 500k–800k tok
    Shipped
    What shipped
    This was most of the actual work: SQL migration off markdown files, the SuperUser/Admin permission split, remote build/merge/rollback dispatch, worktree cleanup, and fixing every real bug that surfaced along the way.
    What was cut
    Compressed from an estimated 5–8 sessions into 3 — the saving here is fewer context reloads, not less rigor.
Where the reduction actually came from

Three phases shipped close to their original scope: intake, the CR store, and the coding-agent orchestration itself — the hardest and most expensive part of the original estimate. Everything cut was a phase that duplicates a check a human is already doing at one of the three gates: a separate test-case generator when the agent already writes its own tests, a PR review when Gate 2 already requires a human click, a staging promotion when nothing has broken production yet.

The two gaps worth closing next are the ones already flagged along the way: a real staging environment (Phase 6), and monitoring that triggers rollback instead of waiting for someone to notice (Phase 10).
